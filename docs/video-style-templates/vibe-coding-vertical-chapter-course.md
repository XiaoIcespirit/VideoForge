# Vibe Coding Vertical Chapter Course

- slug: vibe-coding-vertical-chapter-course
- mode: design-from-brief
- status: draft
- 适合平台: 抖音竖屏优先，可派生为小红书、B 站竖屏切片、视频号短课
- 默认规格: 9:16 竖屏，1080x1920，30fps，每集 3-5 分钟

## 风格定位

把 `tukuaiai/vibe-coding-cn` 中抽象的 Vibe Coding 哲学、方法论和工作流讲成 3-5 分钟能看懂、能复述、能照做的竖屏工程课，整体观感像“AI 工程控制室 + 哲学白板课 + 代码实战拆解”。

资料源分工：

- GitHub README 是系列结构和原始内容的 source of truth，用于确定章节体系、知识范围和项目定位。
- Zread 页面用于单集讲解拆解样例，尤其适合参考“Vibe Coding 哲学原理”这种章节如何从概念、角色分工、范式、工程映射逐层展开。

## 适用场景

- Vibe Coding 哲学原理、核心心智模型、问题求解能力。
- 胶水编程、递归自优化、Canvas 白板驱动开发、AI 蜂群协作。
- Prompt、Skill、Workflow、memory-bank、实施计划、调试与测试等章节。
- 开发环境搭建、AI CLI 使用、从 README 到实操的流程拆解。
- 抽象概念转工程动作的短课，例如“什么叫人想、AI 写、人验”或“为什么文档即上下文”。

不适合把一个大章硬塞进一集。复杂章节应拆成“概念篇 / 方法篇 / 实操篇 / 坑点篇”；简单小节可以合并成一集。

## 叙事结构

固定使用“开场 Hook → 本集问题 → 核心概念 → 通俗类比 → 工程映射 → 小案例/流程演示 → 本集结论”的七段式结构。

每集只服务一个主问题、一个核心概念、一个工程动作。比如“Vibe Coding 是什么”这一集不要同时讲完整工具链，只讲清楚“人描述意图和验证结果，AI 理解意图并执行操作”的协作重构；工具链放到后续集数。

推荐叙事句式：

- Hook：先抛出反常识结论，例如“以后不会写代码，也可能能做软件，但前提是你会验收。”
- 本集问题：把章节标题转成一个观众能立刻理解的问题。
- 核心概念：给出一条清晰定义。
- 通俗类比：把抽象理论换成生活或工程类比。
- 工程映射：落到文件、命令、流程、测试或 memory-bank。
- 小案例：用一个极小动作演示“怎么做”。
- 结论：沉淀成一句可截图的操作原则。

## 镜头节奏

前 5-8 秒快速 Hook，平均 1-2 秒一镜，用大字、对比和动态框选抓住注意力。概念解释段每镜 4-8 秒，允许适当停顿让观众消化。代码、README、流程图或终端演示段每镜 8-15 秒，必须降低画面干扰，保证手机上能看清。

每 45-60 秒出现一次进度提示或小结卡，把观众拉回主线。镜头节奏可按章节难度灵活调整：哲学概念更慢，工具演示更快；但任何时候都不能让单屏同时承载超过一个主概念和两个辅助标签。

内容量控制：

- 3 分钟：1 个概念 + 1 个类比 + 1 个最小工程动作。
- 4 分钟：1 个概念 + 2 个对比点 + 1 个小案例。
- 5 分钟：1 个概念框架 + 1 个流程演示 + 1 个坑点提醒。

## 画面构图

竖屏中心 60% 是主讲解区，上方 12% 只放集数和当前章节标题，不放横杠式进度条，不重复展示系列名。下方 18% 放小雪灵和口播字幕安全区。左右边缘作为“工程注释区”：左侧可放角色、概念标签或路径起点，右侧放工程翻译、命令、文件名或风险提示。

小雪灵 IP 采用“左下静态讲解位”处理：约 240-260px 高的透明贴图，放在字幕框左边，像角色正在说话；不做点头、漂浮、呼吸或场景表演动画。角色不能遮挡标题、主信息或字幕，字幕框需要为角色让出左侧安全区。

章节地图和流程讲解优先使用曲线路径、数据流、终端反馈这类非文字组件表达“进展”和“连接”。中文正文不要长期放在 3D transform、perspective、rotateY 或强 blur 的容器里，避免 Chromium 渲染中文时出现发糊白边。

画面采用三层结构：

- 背景层：深色工程控制室、网格、节点、轻量数据流，不抢主信息。
- 主体层：概念卡、白板推导、代码块、终端或浏览器窗口。
- 标注层：箭头、框选、关键词、状态徽章、章节进度。

横向内容进入竖屏时必须重排：代码块只展示关键片段，README 只截取小区域，流程图使用纵向堆叠或分步展开，不把完整桌面截图硬塞进竖屏。

## 字幕系统

字幕系统分四层：

- 底部口播字幕：每句 10-18 个汉字，采用单行智能字幕条，不做常规两行字幕。小雪灵放在左下后，字幕框右移；字幕必须通过短句切分、字号自动缩放和极端长句单行跟读滑动保持完整可读。
- 中央关键词：只放本屏最重要的 1-3 个词，例如“意图”“验证”“反馈闭环”。
- 工程翻译注释：把抽象词换成动作，例如“描述意图 = 写清目标、对象、约束”。
- 状态标签：用“误区”“做法”“验证”“收敛”“坑点”等短标签标记段落功能。

抽象词必须同步解释。凡是出现“控制论”“概念本体论”“递归自优化”“胶水编程”这类词，必须在同屏或下一屏给出一句通俗翻译，不能只堆术语。

短文字不使用大面积面板。章节问题、金句、警告语和小结优先使用 `bit-blur-slide-word`、`bit-word-by-word`、下划线扫描和小型标签；只有长定义、Prompt、README、代码、终端和流程演示保留内容自适应面板。

字幕跟读滑动只用于单句仍放不下的极端情况，速度跟当前 caption 时长绑定，首尾各保留短暂停顿，避免变成广告跑马灯。关键词高亮优先使用已安装的 remocn `InlineHighlight` 或同类内联高亮组件，不再用大块背景框强调一句话。

## 转场语言

转场服务于“从抽象到工程”的递进感，而不是炫技。

- Hook 到问题：快速 zoom-in 或 hard cut，制造抓力。
- 概念到类比：白板擦写、卡片翻面或 split reveal。
- 类比到工程映射：向下 snap 到文件/命令/流程。
- 工程演示到结论：progress wipe 或 terminal success 状态收束。
- 章节之间：使用纵向滑动或节点跳转，保留竖屏课程的连续感。

Glitch 只能用于“AI 失控、上下文污染、幻觉、复杂性爆炸”等负面示例；平时不要滥用。每集最多使用一种强转场，其余保持克制。

## UI/代码/提示词演示方式

概念先用图谱解释，再落到终端、代码块、README、流程图或 memory-bank 文件结构。演示时遵循“先看结构，再看动作，再看验收”：

- README：用高亮框定位章节标题、关键句和阅读顺序。
- 代码块：只展示与概念相关的关键 5-12 行。
- 终端：只保留命令、关键输出和错误/成功状态。
- Prompt：拆成“目标 / 对象 / 约束 / 验收”四块。
- Workflow：使用纵向步骤条，展示输入、处理、输出、验证。
- Memory Bank：用文件夹树说明上下文如何沉淀。

每个 UI 演示都必须能回答一个问题：“观众看完这一屏，下一步能做什么？”

## 动效语气

动效语气是“专业技术感 + 哲学推导感 + 竖屏短课爽感”。它应该让观众感觉内容很强，但不是浮夸宣传片。

核心动作风格：

- 概念出现：稳、准、有层级，像白板课。
- 工程映射：快速、清晰、有反馈，像开发者工具。
- 坑点示例：短促、带一点错误态冲击。
- 结论卡：停稳、可截图、可复习。

整体避免过度漂浮、过度粒子、过度赛博背景。动效必须帮助理解，不替代理解。

## 色彩与字体方向

色彩以深色工程背景为主，搭配高对比文字。建议方向：

- 背景：近黑、深灰、低饱和冷色网格。
- 主信息：白色或近白。
- 概念强调：青绿色或电蓝。
- 警告/误区：红色或橙红。
- 收敛/成功：绿色。
- 金句/截图卡：黄色或暖白。

字体方向：标题使用粗体、紧凑、现代感强的中文黑体；正文字幕使用高可读无衬线；代码和命令使用等宽字体。不要使用过细字体，不要让深色背景上的低对比灰字承担关键信息。

## 声音与口播假设

默认依赖口播，但静音观看也应能理解 70% 内容。口播语气像“懂工程的人在快速讲明白一件事”，不用过度营销腔。

声音设计：

- Hook：短促低频 hit 或 click-in。
- 概念卡出现：轻 snap。
- 代码/终端执行：键盘、命令确认、轻提示音。
- 坑点/反例：短 glitch 或 error tick。
- 结论卡：稳定落点音，留 1 秒呼吸。

音乐用稳定电子节拍，不要压过口播；每 45-60 秒的小结卡可以跟随音乐节拍做一次轻收束。

## 场景结构建议

| Scene | 时长 | 目的 | 画面动作 | 文字层 | 动效语言 |
| --- | --- | --- | --- | --- | --- |
| Cold Open | 0-8s | 抓住注意力 | 传统开发混乱画面切到 AI 工程控制室 | “不会写代码，也能做软件？” | 快切 + 大字 pop |
| Chapter Map | 8-25s | 告诉观众本集讲什么 | 曲线路径连接 3 个清晰 2D 节点，可叠轻量数据流脉冲 | 本集问题 + 3 个小节 | route draw + node pulse |
| Core Concept | 25-70s | 给出核心定义 | 概念卡逐层展开 | 关键词 + 一句话定义 | blur slide + word reveal |
| Plain-Language Analogy | 70-115s | 降低理解门槛 | 把抽象概念换成生活/工程类比 | “你不是司机，你是导航员/验收员” | card flip |
| Engineering Translation | 115-175s | 把概念落到工程动作 | Prompt 被拆成目标、对象、约束、验收 | “抽象词 → 具体动作” | split reveal |
| Mini Demo | 175-250s | 展示最小可执行流程 | README、终端、文件树或 workflow 逐步演示 | “照着做：第 1 步 / 第 2 步 / 验收” | terminal/code reveal |
| Pitfall Contrast | 250-285s | 提醒误区 | 左右对比“随便聊”与“结构化提问” | “别让 AI 自由发挥到失控” | warning wipe |
| Takeaway Card | 285-300s | 沉淀结论 | 全屏金句卡 + 下集预告 | 一句话原则 + 下一集标题 | hold + subtle pulse |

如果单集只有 3 分钟，合并 `Plain-Language Analogy` 和 `Engineering Translation`，并把 `Pitfall Contrast` 缩短为 10 秒。如果单集接近 5 分钟，保留全部场景，并在 `Mini Demo` 中加入一次更完整的操作演示。

## 复用边界

可复用：

- 竖屏安全区、字幕层级、章节进度条、概念卡样式。
- 七段式叙事结构。
- “抽象概念 → 通俗类比 → 工程动作”的讲解节奏。
- README/代码/终端/流程图的演示规则。
- 已验证组件映射和 remocn 待安装组件方向。

每集必须替换：

- 章节标题和本集主问题。
- 核心概念和通俗类比。
- 工程映射中的具体文件、命令、prompt 或 workflow。
- 小案例和坑点。
- 结论金句与下一集预告。

拆分规则：

- 一个章节如果包含多个核心概念，拆成多集。
- 一个小节如果不足 3 分钟，合并相邻小节或加入一个最小案例。
- 不为凑时长堆术语；优先减少概念量，增加例子和演示。
