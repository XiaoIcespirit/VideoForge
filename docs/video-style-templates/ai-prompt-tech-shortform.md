# AI Prompt Tech Shortform

- slug: ai-prompt-tech-shortform
- mode: design-from-brief
- status: draft
- 适合平台: 抖音、小红书、Bilibili 竖屏切片、YouTube Shorts
- 默认规格: 9:16 竖屏，1080x1920，45-75 秒，30fps

## 风格定位

面向想快速提升 AI 视频质量的创作者，用高密度技术拆解和短视频冲击力，把“提示词控制画面质量”讲成可执行的专业教程。

## 适用场景

- AI 视频教程、提示词技巧、模型原理浅讲。
- before/after 案例拆解。
- 参数控制、prompt 结构、工作流优化。
- 适合把一个复杂概念拆成 3-5 个可复制步骤的短视频。

## 叙事结构

采用“反常识 hook -> 错误示例 -> 原理一句话 -> 五维控制框架 -> 案例对比 -> 可复用公式”的结构。前 3 秒直接抛出反常识结论，中段用失败画面和成功画面对照解释，结尾给出一句可截图的 prompt 公式或检查清单。

## 镜头节奏

前 8 秒平均 1-1.5 秒一切，保持快速抓注意力；原理解释段放慢到 2-3 秒一镜；案例对比段用 0.5-1 秒的局部快切突出差异；结尾公式停留 3-4 秒，给观众截图和复看空间。

## 画面构图

竖屏中心区域放主信息，顶部保留章节/问题标签，底部保留口播字幕安全区。画面以大字卡、prompt 卡片、对比画面和参数面板组成，文本占比可以高，但每屏只允许一个主焦点。左右对比画面要用明确边界，避免信息糊成一团。

## 字幕系统

底部保留短句口播字幕，每句不超过两行；关键概念使用大字关键词，例如“注意力权重”“主体优先级”“动作约束”；逐词高亮只用于公式或关键定义；失败案例使用红色/警告标签，成功案例使用绿色/确认标签；注释文字以小标签贴在画面对象旁边。

## 转场语言

使用快推、局部 zoom、卡片 snap、before/after wipe 和参数面板滑入。概念切换用硬切加轻微 motion blur；案例 reveal 用横向 wipe；不要滥用 glitch，只有在“错误生成”“画面崩坏”时短促使用。

## UI/代码/提示词演示方式

Prompt 展示为卡片或代码块，重点词用 token-chip 拆开。参数解释使用右侧面板或底部 timeline，案例画面放中间。before/after 对照使用左右分屏或上下堆叠，失败点用圈选、箭头和短标签标出。每个操作步骤必须能独立截图理解。

## 动效语气

整体是技术感 + 短视频爆点 + 教程清晰度。动效应该干净、快、明确，不做柔和慢漂浮。所有运动都服务于“指向差异、建立层级、让观众看懂步骤”。

## 色彩与字体方向

背景以深色或低饱和冷灰为主，使用青绿色或亮黄色做技术强调，红色只用于失败/错误状态。标题字体偏粗、清晰、短视频友好；正文字幕保持高对比，不使用过细字体。避免整支视频变成单一蓝紫科技渐变。

## 声音与口播假设

默认依赖口播，但静音也要通过大字关键词和步骤卡看懂 70% 内容。音乐使用稳定电子节拍，前 3 秒可以有强起音效；关键词弹出、参数锁定、案例 reveal 和结尾公式各有一个轻音效点。

## Remotion 组件映射

### 已验证具体组件

- `XiaoxuelingLectureTemplate` from `src/templates/VibeCodingShort.tsx`: 可作为本项目讲解型视频的基础参考，适合 hook、章节地图、核心概念、工程映射、mini demo、坑点和结论等场景结构。
- `LectureScene` scene kinds from `src/templates/xiaoxuelingLecture/types.ts`: 可参考其 `coldOpen`、`chapterMap`、`coreConcept`、`engineering`、`miniDemo` 等类型组织分镜数据。

### 建议组件类型

- Animated keyword caption：用于前 3 秒 hook 和核心概念强调。
- Prompt card / code block：用于展示提示词结构和 token 级拆解。
- Before/after split view：用于失败与成功案例对照。
- Parameter panel：用于展示主体、动作、风格、镜头、约束五类控制维度。
- Timeline progress marker：用于提示教程进度。
- Snap / wipe / zoom transition：用于段落切换和案例 reveal。

## 场景结构建议

| Scene | 时长 | 目的 | 画面动作 | 文字层 | 动效语言 |
| --- | --- | --- | --- | --- | --- |
| Hook | 0-3s | 抛出反常识 | 失败画面快速闪现，成功画面压上来 | “认真写反而翻车？” | 快切 + 大字 pop |
| Failure Contrast | 3-10s | 建立痛点 | 左右对比失败/成功案例 | “不是字数问题，是权重分配问题” | split wipe + 红色标注 |
| Core Principle | 10-20s | 解释核心原理 | Prompt token 被拆成芯片，权重流向主体 | “模型会把注意力分给每个词” | token stagger |
| Five-Dimension Control | 20-42s | 给方法 | 参数面板逐项锁定五个维度 | “主体 / 动作 / 风格 / 镜头 / 约束” | panel snap |
| Case Rewrite | 42-58s | 演示应用 | 错误 prompt 改写成结构化 prompt | “先锁主体，再锁动作” | code highlight |
| Formula Outro | 58-70s | 沉淀模板 | 全屏公式卡 + 下期预告 | “可复用公式：主体 + 动作 + 镜头 + 约束” | hold + subtle pulse |

## 复用边界

可复用：hook 结构、prompt 卡片、参数面板、before/after 对照、公式结尾、字幕层级。必须变化：案例素材、具体 prompt、五维控制项的命名、平台时长、口播密度。长视频课程版应降低快切密度，B 站横屏版应给代码和 UI 更多空间。

## 质量检查清单

- [ ] 前 3 秒是否能无声看懂问题。
- [ ] 每屏是否只有一个主焦点。
- [ ] 底部字幕是否避开平台 UI 安全区。
- [ ] 关键词大字是否没有遮挡案例重点。
- [ ] before/after 差异是否被圈选或标签明确指出。
- [ ] 转场是否服务于段落切换，而不是单纯炫技。
- [ ] prompt 卡片在手机上是否仍可读。
- [ ] 已验证具体组件和建议组件类型是否分开标注。
